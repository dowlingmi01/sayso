Say.SMART is the primary API for the Say.So platform. By default, it utilizes JSON-encoded data to represent requests and responses.

One of the main reasons why Say.SMART was developed was to allow for multiple endpoint calls per api call, to reduce communication overhead and to allow for more elegant front-end code.

Making a Say.SMART request is straightforward: simply send a HTTP POST request to https://app.saysollc.com/ssmart with a variable called data containing the json representation of your request.

$request = ' 12345 abcdef1234567890 survey getAllSurveys ';
$url = “https://app.saysollc.com/ssmart”;
$ch = curl_init($url);
curl_setopt($ch, CURLOPT_HEADER, 0);
curl_setopt($ch, CURLOPT_RETURNTRANSFER, 1);
curl_setopt($ch, CURLOPT_POSTFIELDS, array(“data” => $request));
curl_setopt($ch, CURLOPT_HTTPHEADER, array("Content-Type: application/json"));
$resp = curl_exec($ch); //execute post and get results
curl_close ($ch);

Under Request and Response Data Structures, you can find an overview of the structure for Say.SMART requests and responses. In addition, you can find a sample request and response for each endpoint under the Endpoint sections below.

Typical API Calls

To reach multiple endpoints, requests contain API and Authentication parameters at the top level, along with a requests node. The requests node should contain all the endpoint calls, with each endpoint call given a unique name by the developer (e.g. on the front end code). The same unique names are then referenced in the response, under the responses node. Endpoint calls are processed serially, and by default endpoint processing continues if an error occurs on a prior endpoint (this behavior can be modified). See the Common Request Parameters section below for more details, and note that the Authentication parameters are dictated by user_type (see Panelist Endpoints and Administrator Endpoints for more details).


// the first endpoint request in this call [endpoint class] [endpoint name]
// the second endpoint request in this call [endpoint class] [endpoint name] // "error_code" is non-zero if an API or authentication error has occurred 0
// "error_message" is non-empty if an API or authentication error has occurred
// if no API or authentication errors occurred

// the first endpoint response // non-zero if an endpoint error has occurred 0
// the "errors" node only exists if there are errors [error message 1] [error message 2]
// if no errors, for endpoints that return variables [variable value 1 (value be a node)] [variable value 2 (value be a node)]
// if no errors, for endpoints that return records (note that "records" is an array) 1 ... 2 ...
// pagination info for endpoints that return records 10 54 1 10 6

// the second endpoint response // non-zero if an endpoint error has occurred 0
...
// (see above)
// some endpoints write variables to the top-level common_data object, such as "game" and "notifications".
// For example, if two endpoints are called together, and they both affect the panelist's economy,
// only the second (and most up-to-date) "game" node will be returned in "common_data". [variable value 1 (value be a node)] [variable value 2 (value be a node)]

Single-Endpoint API Calls

Although Say.SMART is designed to allow for multiple endpoint calls per API call, it also provides for a shorthand way to call a single endpoint. When used this way, the Say.SMART responses node is still returned, with a single response within that is always called default. This is equivalent to calling Say.SMART using the above method, with the requests node containing a single request named default.

[endpoint class] [endpoint name] // non-zero if an endpoint error has occurred 0
// other endpoint results are here, see the Response Structure section under Typical API Calls for details.

API Parameters

Below are the top-level API parameters. Authentication parameters are also top-level parameters (i.e. common to all endpoints in the call). Authentication parameters are different for each user_type, and are as such specified under the Panelist Endpoints and Administrator Endpoints sections (public endpoints have no authentication parameters).

Dictates the type of authentication performed and endpoint classes available. Possible values are "public", "administrator" and "panelist" (default). Specifies whether or not processing should continue (to endpoints in the same call that follow) if an endpoint error is encountered.

Authentication Parameters

The panelist's currently active session id, provided by the Public endpoint login.login. The panelist's currently active session key (also provided by login.login).

When the public endpoint login.login is called with a valid email and password combination, it returns a session_id and session_key. Those are then used to authenticate calls to panelist endpoints.

Additionally, any call to a panelist endpoint may return two variables in the common_data node: new_session_id and new_session_key. If those are received, it means that the current session is expiring, and that the newly received session_id and session_key should be used by the client-side code instead. The expiring session becomes fully expired after 15 minutes.
