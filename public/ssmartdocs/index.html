Say.SMART | Say.So's Multipoint API for Research and Tracking

Introduction

Say.SMART is an application programming interface for accessing the Say.So platform.

Authentication

Authentication takes place on two levels.

Api level authentication – Verifies that the session id and session key match and the user is authorized to access the Api.

Endpoint level authentication – The Api has the ability to restrict an individual user’s access to specific endpoints.

Making a call to Say.SMART

Using Say.SMART to make a call is simple: send a POST request to https://app.saysollc.com/ssmart with the variable called "data" holding a json string with your parameters.

$user = 20324;
$key = "sdf2345245sgf";
$data_format = json_encode(array("session_id"=>$user, "session_key"=>$key, "user_type" => "test", "action_class"=>"survey", "action"=>"getAllSurveys"));
$url = “https://app.saysollc.com/ssmart”;
$ch = curl_init($url);
curl_setopt($ch, CURLOPT_HEADER, 0);
curl_setopt($ch, CURLOPT_RETURNTRANSFER, 1);
curl_setopt($ch, CURLOPT_POSTFIELDS, array(“data” => $data_format));
curl_setopt($ch, CURLOPT_HTTPHEADER, Array("Content-Type: application/json"));
$resp = curl_exec($ch); //execute post and get results
curl_close ($ch);

The Api will accept multiple endpoint calls in a single request when utilizing the json via POST method. When making multiple calls in a single request, all requests are wrapped in a top level node called “requests”. Then each request data set is wrapped in an arbitrarily named node which will be returned with the response.

In order to obtain the session_id and session_key, you will make a call to a special endpoints with the user_type of "public". These endpoints do not require a session_id and session_key. They are designed to allow for such functions as logging in, retrieving lost passwords and other actions that would take place pre-authentication.

Pubic endpoints accept various parameters such as email and password for logging in and obtaining the session_id and session_key.

Handling responses from Say.SMART

Response data will be in the format of a json encoded string with the following data structure.

  • Error code (error_code) – The code for the error. Empty if no error. This corresponds to Api level errors. Endpoint level errors are passed in the individual response node. If multiple errors are encountered, the first error is returned. If an Api level error is encountered, no data is returned.

  • Error message (error_message) – The message that corresponds to the error code above.

  • Responses (responses)- The wrapper for the response data.

    • Request name – For single request calls that do not pass a request name, “default” is assigned. For multi request calls, this will be the name of the request as passed. This is used for referencing easily the proper node in multi request responses. There may be multiple instances of this node.

      • Records (records) – Some endpoints return record sets. This could be a list of users for instance. There may be multiple instances of this node.

        • Record key – The key for the record data. This is determined individually for each endpoint. Typically, this is the id of the database row returned. When an id is not available, arbitrary values may be assigned. There may be multiple instances of this node.

          • Record field name – The name of the record field. There may be multiple instances of this node.

            • Record value – The value of the record field.

      • Pagination data name – The name of the pagination field. There may be multiple instances of this node.

        • Pagination data value – The value of the pagination field.

      • Errors returned (errors_returned) – The number of error the endpoint encountered.

      • Errors (errors) – Wrapper for the error messages.

        • Error field name – The name of the error

          • Error value – The value of the error. This may be a multi dimensional array or varying depth.

      • Variables (variables) – Some endpoints may return variables. Variables can consist of a success message for instance.

        • Variable name – The name of the variable. There may be multiple instances of this node.

          • Variable value – The value of the variable.

  • Common data (common_data) – Some endpoints pass data that is not directly related to the endpoint but is necessary due to changes resulting from the endpoint.

    • Common data type – The name of the common data.

      • Common data field name - The name of the common data field. There may be multiple instances of this node.

        • Common data value – The value of the common data field. This may a string or multi dimensional array of varying depth.

Response errors

There are two types of errors:

Api level errors – These prevent the Api from continuing. It can be a failed authentication, poorly formatted request data, or other “serious” errors. No response data is provided beyond the first error code encountered and corresponding error message when an Api level error is encountered.

Endpoint error – These errors may consist of poorly formatted request data, parameter data type validation, inability to retrieve requested data, or other errors that affect only the specific request. If multiple requests are made in the same call, the successful requests are processed and their response is returned unless the continue_on_error setting is set to FALSE.
